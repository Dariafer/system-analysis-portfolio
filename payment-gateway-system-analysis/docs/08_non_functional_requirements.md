# Нефункциональные требования / Non-Functional Requirements

## Производительность / Performance

| Требование | Описание |
|---|---|
| Время ответа API | Запросы на создание и получение платежа должны обрабатываться не дольше 1 секунды при нормальной нагрузке |
| Обработка callback | Результат от банка-эквайера должен обрабатываться не дольше 2 секунд |
| Webhook-уведомления | Webhook должен быть сформирован сразу после изменения финального статуса платежа |

## Надежность / Reliability

| Требование | Описание |
|---|---|
| Идемпотентность | Повторный запрос с тем же `Idempotency-Key` не должен создавать новый платеж |
| Сохранность данных | Все платежи, попытки оплаты, webhook-события и технические логи должны сохраняться в PostgreSQL |
| Повторная отправка webhook | При ошибке доставки webhook система должна сохранять событие для повторной отправки |

## Безопасность / Security

| Требование | Описание |
|---|---|
| API-ключ мерчанта | Каждый запрос мерчанта должен быть связан с конкретным `api_key_hash` |
| Хранение секретов | Секретные значения не должны храниться в открытом виде |
| Проверка входных данных | Все входные параметры должны проходить валидацию |
| Логирование | В логах не должны сохраняться чувствительные платежные данные |

## Масштабируемость / Scalability

| Требование | Описание |
|---|---|
| Рост числа платежей | Структура БД должна поддерживать рост количества платежей и webhook-событий |
| Разделение ответственности | Обработка платежей, webhook и логирование должны быть логически разделены |
| Расширение статусов | Модель должна позволять добавить возвраты, холдирование и частичные списания на следующих этапах |

## Ограничения MVP / MVP Limitations

| Ограничение | Описание |
|---|---|
| Нет реального эквайринга | Банк-эквайер представлен тестовой заглушкой |
| Нет обработки карт | Система не хранит и не обрабатывает данные банковских карт |
| Нет PCI DSS | Соответствие PCI DSS не входит в рамки MVP |
| Нет антифрода | Проверка подозрительных операций не реализуется в MVP |
