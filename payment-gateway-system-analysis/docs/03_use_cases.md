## UC-01 Создание платежа / Create Payment

### Цель / Goal

Создать новый платеж по заказу мерчанта.

Create a new payment for the merchant order.

### Основной актор / Primary Actor

Мерчант / Merchant

### Участники / Participants

| Участник            | Роль                                         |
| ------------------- | -------------------------------------------- |
| Мерчант / Merchant  | Отправляет запрос на создание платежа        |
| Payment Gateway API | Принимает и валидирует запрос                |
| Payment Service     | Создает платеж и управляет бизнес-логикой    |
| PostgreSQL          | Хранит данные платежа и ключ идемпотентности |

### Предусловия / Preconditions

* Мерчант зарегистрирован в системе.
* У мерчанта есть действующий `X-API-Key`.
* Мерчант сформировал уникальный `Idempotency-Key`.
* Заказ создан на стороне интернет-магазина.

### Основной сценарий / Main Flow

| Шаг | Действие                                                      |
| --- | ------------------------------------------------------------- |
| 1   | Мерчант отправляет `POST /api/v1/payments`                    |
| 2   | Payment Gateway API проверяет `X-API-Key`                     |
| 3   | Payment Gateway API проверяет обязательные параметры запроса  |
| 4   | Payment Gateway API проверяет `Idempotency-Key`               |
| 5   | Payment Service создает платеж со статусом `CREATED`          |
| 6   | Payment Service сохраняет платеж в PostgreSQL                 |
| 7   | Payment Service сохраняет ключ идемпотентности                |
| 8   | Payment Gateway возвращает мерчанту данные созданного платежа |

### Альтернативные сценарии / Alternative Flows

| Сценарий                       | Условие                                                         | Результат                                 |
| ------------------------------ | --------------------------------------------------------------- | ----------------------------------------- |
| Некорректный запрос            | Отсутствует сумма, валюта или номер заказа                      | Система возвращает `400 Bad Request`      |
| Неверный API-ключ              | `X-API-Key` отсутствует или недействителен                      | Система возвращает `401 Unauthorized`     |
| Повторный ключ идемпотентности | Такой `Idempotency-Key` уже использовался с теми же параметрами | Система возвращает ранее созданный платеж |
| Конфликт идемпотентности       | Такой `Idempotency-Key` уже использовался с другими параметрами | Система возвращает `409 Conflict`         |

### Постусловия / Postconditions

* Платеж создан со статусом `CREATED`.
* Данные платежа сохранены в PostgreSQL.
* Ключ идемпотентности связан с результатом запроса.

## UC-02 Получение статуса платежа / Get Payment Status

### Цель / Goal

Получить актуальную информацию о платеже.

Get current payment information.

### Основной актор / Primary Actor

Мерчант / Merchant

### Участники / Participants

| Участник            | Роль                       |
| ------------------- | -------------------------- |
| Мерчант / Merchant  | Запрашивает статус платежа |
| Payment Gateway API | Принимает запрос           |
| Payment Service     | Получает данные платежа    |
| PostgreSQL          | Хранит данные платежа      |

### Предусловия / Preconditions

* Мерчант зарегистрирован в системе.
* У мерчанта есть действующий `X-API-Key`.
* Платеж был ранее создан в системе.

### Основной сценарий / Main Flow

| Шаг | Действие                                                       |
| --- | -------------------------------------------------------------- |
| 1   | Мерчант отправляет `GET /api/v1/payments/{payment_id}`         |
| 2   | Payment Gateway API проверяет `X-API-Key`                      |
| 3   | Payment Service ищет платеж в PostgreSQL                       |
| 4   | Система проверяет, что платеж принадлежит данному мерчанту     |
| 5   | Payment Gateway возвращает данные платежа и его текущий статус |

### Альтернативные сценарии / Alternative Flows

| Сценарий              | Условие                                    | Результат                                              |
| --------------------- | ------------------------------------------ | ------------------------------------------------------ |
| Платеж не найден      | `payment_id` отсутствует в системе         | Система возвращает `404 Not Found`                     |
| Нет доступа к платежу | Платеж принадлежит другому мерчанту        | Система возвращает `404 Not Found` или `403 Forbidden` |
| Неверный API-ключ     | `X-API-Key` отсутствует или недействителен | Система возвращает `401 Unauthorized`                  |

### Постусловия / Postconditions

* Статус платежа не изменяется.
* Мерчант получает актуальное состояние платежа.

## UC-03 Обработка результата оплаты / Process Payment Result

### Цель / Goal

Обработать результат оплаты, полученный от заглушки банка-эквайера.

Process the payment result received from the mock acquiring bank.

### Основной актор / Primary Actor

Заглушка банка-эквайера / Mock Acquiring Bank

### Участники / Participants

| Участник                                      | Роль                                             |
| --------------------------------------------- | ------------------------------------------------ |
| Заглушка банка-эквайера / Mock Acquiring Bank | Возвращает результат оплаты                      |
| Payment Gateway API                           | Принимает callback                               |
| Payment Service                               | Обновляет статус платежа                         |
| PostgreSQL                                    | Хранит платеж, попытку оплаты и технические логи |
| Webhook Service                               | Формирует событие для уведомления мерчанта       |

### Предусловия / Preconditions

* Платеж существует в системе.
* Платеж находится в статусе `PROCESSING`.
* Для платежа создана попытка оплаты.
* Заглушка банка-эквайера получила запрос на оплату.

### Основной сценарий / Main Flow

| Шаг | Действие                                                                        |
| --- | ------------------------------------------------------------------------------- |
| 1   | Заглушка банка-эквайера отправляет callback в `POST /api/v1/acquiring/callback` |
| 2   | Payment Gateway API валидирует payload callback                                 |
| 3   | Payment Service находит попытку оплаты по `acquiring_request_id`                |
| 4   | Payment Service определяет результат оплаты                                     |
| 5   | Если результат успешный, платеж получает статус `SUCCEEDED`                     |
| 6   | Если результат неуспешный, платеж получает статус `FAILED`                      |
| 7   | Payment Service сохраняет результат обработки в PostgreSQL                      |
| 8   | Система создает webhook-событие для мерчанта                                    |

### Альтернативные сценарии / Alternative Flows

| Сценарий                     | Условие                                                | Результат                            |
| ---------------------------- | ------------------------------------------------------ | ------------------------------------ |
| Некорректный callback        | Отсутствует `acquiring_request_id` или статус          | Система возвращает `400 Bad Request` |
| Попытка оплаты не найдена    | `acquiring_request_id` отсутствует в системе           | Система возвращает `404 Not Found`   |
| Платеж уже финализирован     | Платеж уже имеет статус `SUCCEEDED` или `FAILED`       | Система не изменяет статус повторно  |
| Недопустимый переход статуса | Callback требует переход, не разрешенный state machine | Система отклоняет изменение статуса  |

### Постусловия / Postconditions

* Платеж получает финальный статус `SUCCEEDED` или `FAILED`.
* Результат оплаты сохранен в системе.
* Создано webhook-событие для уведомления мерчанта.

## UC-04 Отправка webhook-уведомления / Send Webhook Notification

### Цель / Goal

Уведомить мерчанта о финальном результате платежа.

Notify the merchant about the final payment result.

### Основной актор / Primary Actor

Payment Gateway

### Участники / Participants

| Участник                           | Роль                                     |
| ---------------------------------- | ---------------------------------------- |
| Payment Service                    | Создает webhook-событие                  |
| Webhook Service                    | Отправляет уведомление                   |
| Система мерчанта / Merchant System | Принимает webhook                        |
| PostgreSQL                         | Хранит webhook-события и статус доставки |

### Предусловия / Preconditions

* Платеж получил финальный статус `SUCCEEDED`, `FAILED` или `CANCELLED`.
* Для платежа создано webhook-событие.
* В настройках мерчанта задан `webhook_url`.

### Основной сценарий / Main Flow

| Шаг | Действие                                                         |
| --- | ---------------------------------------------------------------- |
| 1   | Webhook Service получает webhook-событие для отправки            |
| 2   | Webhook Service формирует JSON payload                           |
| 3   | Webhook Service отправляет HTTP-запрос на `webhook_url` мерчанта |
| 4   | Система мерчанта принимает webhook                               |
| 5   | Система мерчанта обновляет статус заказа                         |
| 6   | Система мерчанта возвращает HTTP `200 OK`                        |
| 7   | Webhook Service отмечает событие как доставленное                |

### Альтернативные сценарии / Alternative Flows

| Сценарий                   | Условие                                 | Результат                                              |
| -------------------------- | --------------------------------------- | ------------------------------------------------------ |
| Мерчант недоступен         | `webhook_url` не отвечает               | Событие остается в статусе ожидания повторной отправки |
| Ошибка на стороне мерчанта | Мерчант возвращает `5xx`                | Событие сохраняется для повторной отправки             |
| Неверный URL               | `webhook_url` задан некорректно         | Событие получает статус ошибки доставки                |
| Повторная доставка         | Предыдущая отправка завершилась ошибкой | Система повторяет отправку webhook                     |

### Постусловия / Postconditions

* Мерчант получил уведомление о результате платежа.
* Статус доставки webhook сохранен в системе.
* При ошибке доставки событие остается доступным для повторной отправки.

## UC-05 Повторный запрос с Idempotency-Key / Repeat Request with Idempotency-Key

### Цель / Goal

Предотвратить создание дубликата платежа при повторной отправке запроса.

Prevent duplicate payment creation when the request is sent repeatedly.

### Основной актор / Primary Actor

Мерчант / Merchant

### Участники / Participants

| Участник            | Роль                                 |
| ------------------- | ------------------------------------ |
| Мерчант / Merchant  | Повторно отправляет запрос           |
| Payment Gateway API | Проверяет ключ идемпотентности       |
| Payment Service     | Возвращает ранее созданный результат |
| PostgreSQL          | Хранит ключи идемпотентности         |

### Предусловия / Preconditions

* Мерчант уже отправлял запрос на создание платежа.
* При первом запросе был передан `Idempotency-Key`.
* Система сохранила ключ идемпотентности.

### Основной сценарий / Main Flow

| Шаг | Действие                                                                       |
| --- | ------------------------------------------------------------------------------ |
| 1   | Мерчант повторно отправляет `POST /api/v1/payments` с тем же `Idempotency-Key` |
| 2   | Payment Gateway API проверяет `X-API-Key`                                      |
| 3   | Payment Gateway API находит существующий `Idempotency-Key`                     |
| 4   | Система сравнивает параметры нового запроса с параметрами исходного запроса    |
| 5   | Если параметры совпадают, система не создает новый платеж                      |
| 6   | Payment Gateway возвращает ранее созданный платеж                              |

### Альтернативные сценарии / Alternative Flows

| Сценарий                                     | Условие                                                     | Результат                                                                          |
| -------------------------------------------- | ----------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| Параметры отличаются                         | Тот же `Idempotency-Key`, но другая сумма, валюта или заказ | Система возвращает `409 Conflict`                                                  |
| Ключ не найден                               | `Idempotency-Key` ранее не использовался                    | Система обрабатывает запрос как новый                                              |
| Первый запрос завершился до создания платежа | Ключ сохранен, но `payment_id` отсутствует                  | Система возвращает ошибку или безопасно завершает обработку по правилам реализации |

### Постусловия / Postconditions

* Дубликат платежа не создается.
* Мерчант получает предсказуемый результат повторного запроса.
* Система сохраняет консистентность данных.

## UC-06 Отмена платежа / Cancel Payment

### Цель / Goal

Отменить платеж до отправки в обработку банку-эквайеру.

Cancel a payment before it is sent to the acquiring bank for processing.

### Основной актор / Primary Actor

Мерчант / Merchant

### Участники / Participants

| Участник            | Роль                            |
| ------------------- | ------------------------------- |
| Мерчант / Merchant  | Запрашивает отмену платежа      |
| Payment Gateway API | Принимает запрос                |
| Payment Service     | Проверяет возможность отмены    |
| PostgreSQL          | Хранит платеж и технический лог |

### Предусловия / Preconditions

* Платеж существует в системе.
* Платеж принадлежит мерчанту.
* Платеж находится в статусе `CREATED`.
* Платеж еще не отправлен в банк-эквайер.

### Основной сценарий / Main Flow

| Шаг | Действие                                                                            |
| --- | ----------------------------------------------------------------------------------- |
| 1   | Мерчант отправляет `POST /api/v1/payments/{payment_id}/cancel`                      |
| 2   | Payment Gateway API проверяет `X-API-Key`                                           |
| 3   | Payment Service находит платеж                                                      |
| 4   | Payment Service проверяет текущий статус платежа                                    |
| 5   | Если платеж находится в статусе `CREATED`, система устанавливает статус `CANCELLED` |
| 6   | Система сохраняет изменение статуса в PostgreSQL                                    |
| 7   | Payment Gateway возвращает обновленные данные платежа                               |

### Альтернативные сценарии / Alternative Flows

| Сценарий                  | Условие                                             | Результат                             |
| ------------------------- | --------------------------------------------------- | ------------------------------------- |
| Платеж не найден          | `payment_id` отсутствует в системе                  | Система возвращает `404 Not Found`    |
| Платеж уже обрабатывается | Платеж находится в статусе `PROCESSING`             | Система возвращает `409 Conflict`     |
| Платеж уже финализирован  | Платеж находится в статусе `SUCCEEDED` или `FAILED` | Система возвращает `409 Conflict`     |
| Неверный API-ключ         | `X-API-Key` отсутствует или недействителен          | Система возвращает `401 Unauthorized` |

### Постусловия / Postconditions

* Платеж получает статус `CANCELLED`.
* Платеж не отправляется в банк-эквайер.
* Изменение статуса фиксируется в системе.
